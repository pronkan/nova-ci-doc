# 3. Security and Compliance Architecture

Nova is fundamentally designed with **Zero-Trust Principles**, operating under the assumption that all executed user code (pipelines) is potentially hostile and untrusted. The platform employs defense-in-depth to isolate execution and protect the cluster.

These robust security measures are explicitly designed to help organizations achieve and maintain **SLSA Level 4** compliance.

## Core Security Pillars

### 1. Cryptographic Identity (SPIFFE/SPIRE)
Nova completely eliminates manual token management (e.g., K8s ServiceAccount tokens, hardcoded secrets).
*   **Workload Attestation:** Every component (Coordinator, Controller, Agent, Plugins) receives a cryptographic identity via SPIFFE/SPIRE.
*   **Mutual TLS (mTLS):** All inter-component communication (e.g., Agent <-> Controller, Controller <-> Telemetry Router) is secured and authenticated using mTLS.

### 2. Network Isolation (L4 & L7)
The execution sandbox (`ci-run-XXXX`) is heavily guarded at the network level.
*   **Layer 4 (K8s Network Policies):** Blocks unauthorized IP traffic. The Agent cannot reach the K8s Kubelet, cloud metadata endpoints, or other namespaces.
*   **Layer 7 (Istio AuthZ):** Enforces strict RPC-level rules based on SPIFFE identity. The Agent is explicitly denied access to the internal Control Plane API and can only talk to its assigned Controller.

### 3. Real-Time Data Loss Prevention (DLP)
The Controller acts as a mandatory proxy for all outbound data.
*   **Aho-Corasick Secret Masking:** An algorithmic engine deployed directly on the Controller. It scans streaming logs (`stdout`/`stderr`) using an Aho-Corasick pattern matching tree to identify and redact injected secrets in microseconds, ensuring they never persist.
*   **PVC Courier:** The Controller handles all artifact transfers. The Agent cannot arbitrarily mount volumes or write data outside the namespace. 

### 4. Supply Chain Security (SLSA Level 4)
Nova enforces strict governance over what code is allowed to run and how artifacts are recorded.
*   **`NovaRepository` CRD:** Defines explicit, vetted code sources, allowed branches, and permitted credentials. Webhooks from untrusted repositories are dropped at the API Gateway.
*   **Provenance Attestation:** The Controller acts as a trusted signer, cryptographically attesting that specific artifacts were generated by a specific run, storing this provenance data within the ArangoDB Graph.

## Governance

### `NovaPolicy` CRD
Security administrators define cluster-wide security rules using the `NovaPolicy` CRD.
*   **Hierarchical Overrides:** Defines default network access (e.g., internet egress blocked by default) and telemetry limits.
*   **Job-Specific Exceptions:** Certain jobs can request overrides (e.g., needing to push a Docker image), which must be explicitly allowed by the overarching policy.
